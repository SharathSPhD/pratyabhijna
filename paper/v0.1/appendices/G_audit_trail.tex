\section{Audit Trail}\label{app:audit}

The single most important architectural commitment we make in this paper is that every numerical claim in the body is derived from a JSON or text artefact under \texttt{audit/} or \texttt{benchmarks/results/} that was produced by code in \texttt{src/} and is reproducible from the recipe in Appendix~\ref{app:reproducibility}. The mapping is:

\begin{itemize}[leftmargin=1.2em]
\item \textbf{Operator behaviour} $\rightarrow$ \texttt{tests/operators/test\_*.py} (passing tests are the ground truth for the equations in Appendix~\ref{app:operator_specs}).
\item \textbf{Plugin smoke} $\rightarrow$ \texttt{audit/phase8/verify\_artifact.json} (15/15 MCP tools exercised in-process).
\item \textbf{Per-arm raw outputs and per-axis scores} $\rightarrow$ \texttt{benchmarks/results/\{poetry\_gen,poetry\_interp,aut,sci\_creativity\}.json}.
\item \textbf{Pre-registered hypothesis tests} $\rightarrow$ \texttt{benchmarks/results/stats.json}.
\item \textbf{Figures} $\rightarrow$ \texttt{paper/figures/*.png} and \texttt{presentation/figures/*.png} (both generated from the same call to \texttt{benchmarks/figures.py}).
\item \textbf{Auto-generated Table~\ref{tab:primary}} $\rightarrow$ \texttt{paper/autoreport.tex} (regenerated by \texttt{benchmarks/autoreport.py} after stats finish).
\end{itemize}

\subsection{What ``no-stub'' means here}

The \texttt{ralph-loop} completion-promise spine adopted from Anthropic's plugin gallery enforces a hard constraint: no stubbed function, no mocked output, no fabricated number. A function that cannot fulfil its contract raises; a benchmark item that cannot be scored is recorded with \texttt{error} set and propagated through the statistical pipeline as missing data (paired observations require both arms to be present). The Phase~5 ``anti-stub'' gate (\texttt{audit/phase5/anti\_stub.json}) enumerates every operator file and every test file that participates in this contract.
