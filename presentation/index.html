Pratyabhijñā Creative Engine

A recursive self-reflexivity layer for LLM creative cognition: the five-śakti cascade as an executable Claude Code plugin, gated by Bayesian Model Reduction. v0.3 directly addresses the four falsifiable findings of the v0.2 adversarial review: (i) clean Haiku CLI substrate (subprocess flag isolation + scrubbed HOME + per-item integrity probe); (ii) event-gated commit on jñāna ΔF with always-shadow revision; (iii) full active-inference uplift on the cascade causal path; (iv) two new control arms isolating the architecture from the compute and protocol confounds.

Motivation

Large language models routinely produce competent text but rarely produce text whose interpretive multiplicity is deliberately authored. Where a single line of poetry can sustain two simultaneously-coexisting readings (Wittgenstein's aspect-shift), LLMs typically collapse onto the dominant reading. We argue that the missing structural element is a recursive self-reflexivity layer: an explicit operator that re-examines its own surface against the same constraint manifold that produced it.

Abhinavagupta's Pratyabhijñā school developed an unusually fine-grained taxonomy of exactly this layer (the five śaktis and the recursive vimarśa loop). Apohana-śakti (contrastive exclusion) is precisely the geometric move that Bayesian Model Reduction (Friston & Penny, 2011) formalises as posterior switching under a parameter-pruning prior. We make this isomorphism concrete and testable.

The seven operators

1. Cit · cit(prompt, τ, seed)

Single-sample LLM generation; the luminous ground.

2. Ānanda · ananda(c, κ, R)

Aesthetic-coherence reward over (constraint-fit, retrieval diversity, form fidelity).

3. Icchā · iccha(prompt, K, κ)

K candidate generations under a constraint-injected prompt.

4. Apohana · apohana({c}, must_avoid)

Contrastive exclusion: 1 − max_j cos(φ(c_i), φ(m_j)).

5. Jñāna · jnana(c, a) → (i*, ΔF)

Posterior selection via Bayesian Model Reduction over Dirichlet-multinomial pseudo-counts.

6. Kriyā · kriya(c, mode)

Surface enaction (verbatim / polish / claude_polish).

7. Vimarśa · vimarsa(surface, aspects, R) → (event, novelty)

Recursive aspect-shift detector. Fires if any of: aspect-multiplicity (≥ 2 aspect-targets each cosine ≥ 0.40), low novelty (max retrieval-cosine ≥ 0.85), or non-negative residual ΔF. This is the layer the paper measures.

Benchmark protocol (v0.3 four-arm matrix)

Four Haiku arms per item, all routed through the v0.3 clean CLI substrate (subprocess flag isolation + scrubbed HOME + per-item IntegrityProbe): haiku_bare (no PCE, primary control), haiku_cascade (PCE event-gated cascade, primary treatment), haiku_bare_2K_scorer (1 pass with K'=2K candidates fanned out by icchā + jñāna BMR scorer; extra-compute control, H6.v3), haiku_generic_revise_2pass (2 passes with a fixed generic revise brief replacing the vimarśa brief; revision-protocol control, H7.v3). Items are pre-registered in benchmarks/items.py (same v0.2 sample); scoring is deterministic in benchmarks/scoring.py; the v0.3 stats / per-domain JSON live under benchmarks/results_v0.3/.

Discussion (v0.3)

v0.3 directly addresses the four falsifiable findings of the v0.2 adversarial review. The headline contrast shifts from "did the cascade beat bare?" (H1.v3–H4.v3, primary) to "did the cascade beat more compute on the same substrate?" (H6.v3, the headline fairness test). H7.v3 isolates the content of the vimarśa brief from the existence of a 2-pass revision protocol, and H8.v3 verifies the cascade's revision pass actually outscores its own draft on items where the event-gated commit chose revision. Length-controlled scoring addresses the v0.2 review's worry that the cascade's longer surfaces alone might explain the win.

The v0.3 architectural changes are causally specific and traceable to the v0.2 review findings: clean Haiku CLI substrate (ADR-001) eliminates context-leakage of the bare control; event-gated commit + always-shadow revision (ADR-002) lifts vimarśa from prompting to a causal step on the jñāna ΔF signal; aspect-conditioned BMR (ADR-003), Hopfield warm-start in apohana (ADR-004), and per-item free-energy budget (ADR-005) bring active inference onto the causal path that v0.2 left as prompting.

Reproducibility

git clone https://github.com/SharathSPhD/pratyabhijna.git
cd pratyabhijna
uv venv && uv sync --extra dev
uv run python scripts/verify_real_model.py
uv run pytest -m "not slow"

# v0.3 outer-host smoke (must pass before any inner-substrate hardening):
uv run python scripts/verify_outer_host_loads_pce.py

# v0.3 prove-gate (single-case validation; per-item integrity probe + leakage scan):
uv run python scripts/prove_gate.py --strict

# v0.3 pilot (~$8-10 Haiku envelope under the locked $20 cap):
uv run python benchmarks/driver.py --pilot --out-dir benchmarks/results_v0.3 --cost-cap-usd 20
uv run python benchmarks/stats.py --results-dir benchmarks/results_v0.3 --out benchmarks/results_v0.3/stats.json
uv run python benchmarks/figures.py --results-dir benchmarks/results_v0.3 --stats benchmarks/results_v0.3/stats.json --out-dirs paper/figures
uv run python benchmarks/autoreport.py --stats benchmarks/results_v0.3/stats.json --paper-dir paper

Every numerical claim on this page is derived from benchmarks/results_v0.3/stats.json and the per-domain JSON files; the v0.3 audit lives under audit/v0.3/ with per-item integrity probes in audit/v0.3/integrity_probes.jsonl, per-run cost snapshots in audit/v0.3/cost_snapshot.json, and the pilot run log at audit/v0.3/pilot_run.log.
